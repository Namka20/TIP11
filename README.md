## Ответы на контрольные вопросы

**1. Что такое GraphQL и в чём его основное отличие от REST API?**
GraphQL — язык запросов к API, где клиент сам определяет структуру и нужные поля ответа. Отличие от REST: один endpoint, нет over-fetching/under-fetching, гибкие запросы вместо жёсткой структуры ресурсов.

**2. Для чего используется GraphQL-схема?**
Определяет типы данных, доступные запросы (Query), мутации (Mutation) и подписки, служит контрактом между клиентом и сервером.

**3. Чем Query отличается от Mutation?**
Query — для чтения данных (без побочных эффектов). Mutation — для изменения данных (POST/PUT/DELETE), выполняется последовательно и гарантирует применение изменений.

**4. Что такое резолвер и какую роль он выполняет?**
Функция на сервере, которая знает, как получить данные для конкретного поля схемы — из БД, REST API, кэша или другого источника.

**5. Почему GraphQL позволяет уменьшить over-fetching данных?**
Клиент указывает только нужные поля в запросе, сервер возвращает ровно их, без лишних данных (в отличие от REST, где endpoint часто отдаёт фиксированную структуру).

**6. Для чего используется библиотека gqlgen в Go-проекте?**
Генерирует типизированный код на Go на основе GraphQL-схемы (резолверы, модели), обеспечивает compile-time безопасность и производительность.

**7. Почему желательно использовать общий сервисный или репозиторный слой для REST и GraphQL?**
Чтобы не дублировать бизнес-логику и работу с БД — и REST, и GraphQL вызывают одни и те же сервисы/репозитории.

**8. Что произойдёт, если GraphQL-запрос попытается получить слишком большой объём данных?**
Сервер может столкнуться с высокой нагрузкой. Нужно внедрять пагинацию, лимиты глубины запроса, ограничения на количество запрашиваемых записей.

**9. Какие преимущества даёт Playground при тестировании API?**
Интерактивная IDE для написания и тестирования запросов, автодополнение, просмотр документации, отладка без внешних инструментов (curl/Postman).

**10. В каких случаях GraphQL особенно удобен на практике?**
- Сложные клиентские интерфейсы с разными наборами данных
- Агрегация данных из нескольких источников
- Часто меняющиеся требования к API (без версионирования)
- Мобильные приложения с ограниченным трафиком
